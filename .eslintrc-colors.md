# Color Usage Validation

## ESLint Rule: No Hardcoded Colors

This project enforces semantic color usage through an ESLint rule that prevents hardcoded Tailwind color values.

### ❌ NOT ALLOWED

```tsx
// Direct color classes
<div className="text-blue-500">Text</div>
<div className="bg-green-100">Background</div>
<div className="border-red-600">Border</div>
```

### ✅ ALLOWED

```tsx
// Semantic color tokens
<div className="text-info">Text</div>
<div className="bg-success/10">Background</div>
<div className="border-destructive">Border</div>

// Theme-based colors
<div className="text-primary">Text</div>
<div className="bg-secondary">Background</div>
<div className="border-border">Border</div>
```

## Available Semantic Tokens

### Status Colors
- `success` - Green for positive states
- `warning` - Orange/yellow for warnings
- `info` - Blue for informational
- `destructive` - Red for errors/negative states

### Theme Colors
- `primary` - Brand primary color
- `secondary` - Secondary surfaces
- `accent` - Accent highlights
- `muted` - Muted/disabled states
- `foreground` - Main text
- `background` - Main background

### Contact Colors (WhatsApp)
- `whatsapp` - WhatsApp brand green
- `contact-sales` - Sales team
- `contact-service` - Service team
- `contact-owner` - Owner direct

## Opacity Modifiers

Standard opacity levels: `/10`, `/20`, `/30`, `/40`, `/50`, `/75`

```tsx
<div className="bg-success/10">10% opacity</div>
<div className="border-warning/20">20% opacity</div>
```

## Exceptions

### Button Component WhatsApp Variant
The `ui/button.tsx` file contains an intentional exception for WhatsApp brand colors in the button variant. This is acceptable for brand recognition.

## Checking Your Code

Run ESLint to check for color violations:

```bash
npm run lint
```

Fix automatically where possible:

```bash
npm run lint -- --fix
```

## Pre-commit Hook

Consider adding a pre-commit hook to prevent hardcoded colors:

```json
{
  "husky": {
    "hooks": {
      "pre-commit": "npm run lint"
    }
  }
}
```
